## Hypothesis

The quality improvement felt in Lab 4 shows up as measurable numbers in Ragas metrics, and the per-metric breakdown localizes failure to a specific pipeline stage — retrieval, ranking, or generation.

## Background

RAG evaluation uses four stage-specific metrics:

| Metric | Question it answers | Failure it diagnoses |
|--------|--------------------|--------------------|
| **Context precision** | Of the retrieved chunks, how many were relevant? | Too much noise retrieved |
| **Context recall** | Of all relevant chunks that exist, how many were retrieved? | Relevant chunks missed |
| **Answer faithfulness** | Is the answer grounded in the retrieved context? | Model hallucinated beyond context |
| **Answer relevance** | Does the answer actually address the question? | Answer is correct but off-topic |

These metrics turn "the answer was bad" into a specific, actionable diagnosis. A low context recall means fix the retriever. Low faithfulness means the model is ignoring the context — tighten the prompt or lower temperature. Low context precision means the retriever is pulling noise — improve chunking or switch embedders.

## Expected Observations

- **C scores highest overall** — reranking improves context precision, compression improves answer relevance by reducing distraction.
- **A shows low context precision** — the unranked dump includes irrelevant chunks.
- **The per-metric breakdown pinpoints failure:** if A has good context recall but low faithfulness, the retriever found the right chunks but the model ignored them — a prompt structure problem, not a retrieval problem.
- The metric differences between A and C should be larger than between B and C — structure helps more than ordering alone.

## Why This Matters

RAG evaluation is its own engineering discipline. A single "quality score" hides whether the problem is in the retriever, the ranker, or the generator. Once you can read Ragas output diagnostically, you can identify exactly which component to improve rather than tuning the whole pipeline at once. This is also the foundation for regression testing: run the eval suite before and after any pipeline change to confirm improvement and catch regressions.
